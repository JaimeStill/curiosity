**Culled**: sparsesetmap from forward implementation scope. The two-variant comparison (map vs slice sparse representation) was an instrumentation strategy to isolate the sparse-mapping representation's contribution to iteration cost; with that contribution now quantified (sparsesetslice within ~1% of archetype across 1k/10k/100k; sparsesetmap with cache-cliff growth from 18.87 → 23.95 → 31.35 ns/entity), the instrument has served its purpose. The map variant does not represent a production sparse-set design — real engines (EnTT and similar) use paged sparse arrays, not hash maps. Forward decision: sparsesetmap's six stubbed methods will not be filled, it will not run new workloads, and it will not appear in subsequent result tables beyond the iteration baseline. The source is preserved this session as the implementation behind the captured measurements; per D-012 it is removed when the experiment as a whole is removed. Subsequent work targets archetype + sparsesetslice + sparsesetgroup — three backends.
