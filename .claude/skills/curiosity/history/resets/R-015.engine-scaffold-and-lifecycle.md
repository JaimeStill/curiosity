**Workflow observations this session**:

- **Plan-mode at session start surfaced two structural questions
  before exit.** Herald source location and Coordinator-adaptation
  strategy were both load-bearing for the first source-code
  checkpoint. Resolving them via AskUserQuestion in plan mode kept
  the development phase clean. Pattern: questions that bear on the
  first source-code checkpoint should be asked at plan exit, not
  surfaced mid-checkpoint.

- **A wrong-assumption question got rejected and the user clarified
  the real intent.** The module-path question I offered pre-framed
  the answer around D-028's sibling-repo model; the user interrupted
  to clarify the in-tree prototype-phase intent. The interruption
  was faster than executing the wrong assumption. Pattern:
  structured-option questions can pre-frame the user's answer to
  fit my prior assumptions; the user's interruption is the recovery
  mechanism, and the cost pre-empts a wasted source-code phase.
  When in doubt, default the framing to the *current observable*
  state (`.gitignore` had `/engine/`, but no `engine/` directory
  existed yet — that ambiguity should have triggered a more open
  question, not an option list).

- **Reorder of source-code checkpoints surfaced from dependency
  direction.** The initial plan put `engine.go` before
  `lifecycle.go`; the dependency direction (Engine holds a
  `*lifecycle.Coordinator`) pointed the other way. Source-code.md's
  "Inner types and helpers before outer types that compose them"
  applied directly. Pattern: anticipated checkpoints in the plan
  file are forward commitments that adjust as work progresses;
  reorders justified by published phasing guidance don't warrant
  re-approval.

- **Naming evolution is user-side activity the checkpoint workflow
  tolerates well.** Three rename events across the session:
  `WaitForStartup` → `Startup`, `Startup` → `Start`, `OnStartup` →
  `OnStart`. Each was user-initiated mid-session; my job was
  acknowledging, identifying the follow-on call-site edit in
  `engine.go`, and continuing. Pattern: source-code naming belongs
  to the user; my role is tracking implications across files, not
  relitigating the choice.

- **Hot-start test scope adjusted on user feedback about
  brittleness.** The initial test asserted readiness immediately
  after `Start()` — correct against the current synchronous
  Coordinator but tightly coupled to that specific shape. User
  flagged it as "destined to have issues" and asked to follow the
  real readiness pattern or defer. The pattern is undesigned (it
  resolves with the frame-loop adaptation framework), so the test
  was trimmed to settled behavior. Pattern: tests should follow the
  real consumer pattern; when that pattern is undesigned, deferral
  is more honest than locking in a shape that pre-decides the
  question.
