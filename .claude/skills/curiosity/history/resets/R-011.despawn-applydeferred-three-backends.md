**Integrated**:
- `experiments/ecs-storage-comparison/archetype/archetype.go` ‚Äî
  `Storage` gains a `despawned []entity.ID` queue. `Despawn(id)`
  appends to the queue. `ApplyDeferred()` walks the queue calling a
  private `applyDespawn(id)` per id, then truncates `[:0]` to preserve
  capacity. `applyDespawn` looks up the entity's `(arch, row)` via
  `s.locations[id]`, swap-removes the row across `arch.entities` and
  every column's `data` byte buffer (cross-column row alignment
  invariant preserved in lockstep), updates `s.locations[movedID]` to
  the swapped-in entity's new row, shrinks `arch.entities` and each
  column's `data` by one row, deletes `s.locations[id]`, and calls
  `s.alloc.Free(id)`. No defensive `if !ok` on the location lookup
  (panic-natural per the session's design discussion).
- `experiments/ecs-storage-comparison/sparsesetslice/sparsesetslice.go`
  ‚Äî `Storage` gains `despawned []entity.ID`. `Despawn` / `ApplyDeferred`
  shapes mirror archetype's. `(*Storage).applyDespawn(id)` iterates
  `s.columns` and probes `int(id) < len(c.sparse) && c.sparse[id] != -1`
  per column, calling `(*column).swapRemove(id)` for each column the
  entity is in, then `s.alloc.Free(id)`. New `(*column).swapRemove`
  method does the per-column swap-remove: lookup `r := c.sparse[id]`,
  if `r != lastRow` swap `c.entities[r] ‚Üî c.entities[lastRow]` + copy
  dense bytes + `c.sparse[movedID] = r`; shrink `c.entities` and
  `c.dense`; `c.sparse[id] = -1`. The scan-all-columns
  membership-lookup choice (vs. a top-level signatures map) is logged
  as D-025.
- `experiments/ecs-storage-comparison/sparsesetgroup/sparsesetgroup.go`
  ‚Äî `Storage` gains `despawned []entity.ID`. `Despawn` / `ApplyDeferred`
  same shape. `(*column).swapRemove` identical to the sparsesetslice
  version (column-package-local types; same code, distinct package).
  `(*Storage).applyDespawn(id)` runs in two phases. Phase A iterates
  `s.groups`; for each group, picks `c0 := g.columns[0]` as the
  prefix-membership probe, lookups `r := c0.sparse[id]`, gates on
  `r >= 0 && r < int32(g.size)`. If the entity is in `g`'s prefix and
  not already at `boundary := g.size - 1`, lockstep-swaps row `r` ‚Üî
  `boundary` across every column in `g.columns` (entities slot,
  `swapDenseRows`, `c.sparse[id] = boundary`, `c.sparse[other] = r`,
  with `other` hoisted from `c0.entities[boundary]` since lockstep
  guarantees it's the same across all group columns); decrements
  `g.size`. Phase B is identical to sparsesetslice's
  scan-and-swap-remove sweep, then `s.alloc.Free(id)`. Multi-group /
  shared-column lockstep correctness is deliberately out of scope ‚Äî
  both workloads declare a single group, so the case isn't exercised.

**Workflow observations this session**:
- **Per-backend design discussions before each source checkpoint
  kept pacing aligned.** Cross-backend patterns (command-buffer
  shape, dead-id semantic) settled once for archetype and propagated
  implicitly to the others; backend-specific patterns
  (sparsesetslice's membership lookup, sparsesetgroup's lockstep
  coordination scope) surfaced their own questions only when the
  variable was new. Pattern: ask only when the backend introduces
  a question the prior backends didn't.
- **`sparsesetgroup` source phasing ‚Äî split into infrastructure +
  lockstep checkpoints ‚Äî respected the screen budget and gave the
  lockstep prep the focused moment it earned.** Pattern: when a
  backend has a novel coordination layer beyond the shared shape,
  separate the shared shape from the novel layer at the checkpoint
  boundary.
- **User nudge "just go vet is fine" was not new feedback ‚Äî D-020
  already establishes this.** Pattern: check the decisions log
  before treating a brief operational nudge as new guidance, to
  avoid decision-log churn where the rule is already established.
- **Multi-group / shared-column lockstep correctness was deliberately
  out of scope.** Both existing workloads declare a single group, so
  the case isn't exercised. The Despawn surface assumes single-group
  semantics; a future workload using multiple groups (none currently
  planned) would need this surface revisited before running.
