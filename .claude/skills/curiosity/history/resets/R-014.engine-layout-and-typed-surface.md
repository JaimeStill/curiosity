**Workflow observations this session**:

- **Plan-mode closeout review caught a naming conflict before
  artifacts landed.** During the closeout planning discussion (in
  plan mode), reading the prior decisions log surfaced D-017's
  "Task" terminology decision, which conflicted with a session-
  level statement made earlier in the day. Asking the user to
  reconcile *before* writing decisions prevented a contradictory
  D-### entry. Pattern: closeout plan-mode review should walk the
  prior decisions log for any prior commitments the session's
  outcomes touch — not just for orientation at session start.

- **Struct-based view emerged late as the better Query design.**
  Phase B's Query discussion initially anchored on per-count
  generics (`Query1`, `Query2`, ..., `QueryN`); the struct-based
  view alternative was only offered after the user pushed back
  on suffix proliferation and asked about Go 1.26 features. The
  alternative was clearly stronger on ergonomics with negligible
  performance cost. Pattern: when an API design dimension hits a
  language constraint (here, no variadic type parameters), survey
  the workable alternatives *before* recommending one — don't
  anchor on the first option that fits the constraint.

- **Knowledge-cutoff-sensitive claims verified at decision time.**
  The user's question about Go 1.26 features triggered WebFetch
  research against the official release notes. The findings (1.25
  had no language changes; 1.26 added two unrelated features)
  closed the door on language-level relief and validated the
  struct-based view recommendation. Pattern: claims about language
  or framework features past the assistant's training cutoff need
  explicit verification when they bear on a decision; don't assert
  from training memory.

- **Session scope held as a forward commitment.** The session-
  start scope question (which of R-013's three deliverables to
  attempt) explicitly deferred deliverable 3, leaving room for
  depth on the layout + typed surface work. By session end the
  Phase A and Phase B discussions had each absorbed substantial
  attention; attempting all three would have rushed at least one.
  Pattern: scope as a forward commitment that adjusts as work
  progresses (per execution.md §Session start) is a working
  principle, not just a procedure. The deferred deliverable
  folds naturally into the next session(s) without losing its
  thread.

- **D-023's carry-forward pattern from experiment to engine
  worked exactly as intended.** D-023 (per-primitive low-level
  packages) was originally written for the experiment's
  restructuring but explicitly named "the pattern is intended to
  carry into engine architecture." Phase A's package decomposition
  discussion confirmed the pattern at the engine layer with no
  reframing needed. Pattern: decisions written with explicit
  forward-applicability text save downstream sessions the work
  of re-deciding; the explicit "intended to carry" framing was
  load-bearing here.
