# R-010 — package restructure into entity / component low-level packages, plus bare-minimum entity allocator — 2026-05-08

**Scope**: project + engine

**Trigger**: Carryout of R-009's "Next session focus" — implement
`Despawn` + `ApplyDeferred` and the structural-churn workload across
the three active backends. Mid-orientation, the user surfaced that
without entity-ID recycling the structural-churn measurements would
not reflect how the storage implementations behave in practice
(`concepts/engine/entity-allocator.md` flagged the same risk under
*Recycling queue* and *Allocator above storage layer*). The user
further redirected my initial allocator placement to surface a
deeper layering principle: each core ECS primitive should own its
own low-level package, establishing clean per-primitive ownership
before more code accretes. The two adjustments together absorbed
the session — deliverables 2 and 3 from R-009's focus carry forward.

**Workflow observations this session**:
- **Mid-orientation scope-shift was load-bearing twice.** The user
  surfaced (a) the entity-allocator dependency on measurement
  validity, then (b) the per-primitive package layering. Both
  arrived during the orientation phase before any source code
  landed; both reshaped the session's deliverables. The plan file
  reflected each shift in turn rather than persisting a stale
  agenda. Pattern: when scope-shift signals arrive in orientation,
  treat them as primary input and revise the plan in place — the
  cost of a new orientation is small versus carrying a misaligned
  agenda into development.
- **Auto-mode authorization for mechanical migration was the right
  call.** After the package restructure was agreed and the rename
  surface was unambiguous, the user authorized executing the
  migrations directly rather than scene-setting each. This
  collapsed many per-checkpoint cycles for content the user had
  already approved at the design level. The substantive change
  (allocator wiring) happened inside the same execution pass,
  blurring the source-code-authorship boundary in a way that
  worked for a mechanical-mostly migration — but it should be
  read as a session-specific authorization, not a general pattern.
- **Local-variable shadow surfaced from package introduction.**
  Three iterator files used `entity` as a local variable; the new
  `entity` package created a shadow that compiled but read badly.
  Renamed to `eid` during the migration. Pattern: introducing a
  new low-level package can silently shadow common variable names
  in dependent code; sweep dependent files for shadowing during
  package-introduction migrations.
