**Workflow observations this session**:
- **Workload + four interface methods × three backends landed in 19
  checkpoints across two workloads.** The shared shape across backends
  (deferred-queue infra → Read/Write → push-side Attach/Detach → apply
  helpers + drain) let each backend's implementation reuse the prior
  backend's pattern; pacing held at one checkpoint per scoped concern
  with screen budget respected throughout.
- **Mid-session question about culling sparsesetgroup before
  implementing attach/detach** — engaged before the lockstep
  maintenance code landed. Decision was to complete with eyes open
  (the sparsesetmap R-007 cull analogy didn't apply because
  sparsesetgroup isn't structurally dominated; the boundary-cross
  cost was an unmeasured question, not a settled one). Data validated
  the call: sparsesetgroup pays no detectable boundary-cross penalty
  (~0.1% from sparsesetslice on attach/detach). Pattern: culls based
  on inspection require structural dominance; "complete and measure"
  is cheaper than "speculate and cull" when the cost question is open.
- **Helper factoring in archetype (`swapRemove`) and sparsesetgroup
  (`enterPrefixIfCovered`, `exitPrefixIfInPrefix`)** was justified
  by the screen budget (inline alternatives pushed applyAttach /
  applyDetach over 50 lines) and the named-operation clarity (the
  prefix entry/exit operations are conceptual peers to what Spawn
  and applyDespawn already do inline). Defensible against the
  project's "no premature abstractions" rule because the helpers are
  doing real work, not anticipatory generalization.
- **The Read/Write map-lookup tax surfaced as the dominant cost in
  attach/detach** (~13 ns/op × 200k ops/frame ≈ 2.6 ms gap at 100k).
  This was not predicted by prior workloads — the iterator-based
  workloads cached the column pointer once per Query and never paid
  the per-call lookup. The attach/detach workload was the first to
  exercise the per-entity point-Read/Write surface at scale. Forward
  implication noted in D-027: the engine's typed call-site surface
  design will want this finding in scope.
- **Mid-session workload completion** — R-012's plan was attach/detach
  only; the user opted to push mixed into the same session to settle
  D-027 in one closeout. The decision was made when context was warm
  on the attach/detach analysis (the same analytical frame applied
  to mixed). Pattern: completing a multi-workload row's last entry
  in the same session as its penultimate makes sense when the
  intermediate analysis is the gating context, not when fresh
  measurement context is needed.
