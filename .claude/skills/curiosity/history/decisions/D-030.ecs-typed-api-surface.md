# D-030 — ECS typed API surface — 2026-05-12

**Scope**: engine

**Context**: D-029 settled that the engine's typed call-site surface
goes directly to archetype with no internal type-erased layer.
What that surface looks like — the signatures and semantics of
Spawn, Despawn, Attach, Detach, Get, Set, Has, and Query — was the
balance of Phase B in this session. D-016's task declaration surface
forces read/write per component at the API; D-024's deferred-mutation
discipline forces structural mutations to defer to safe sync points;
D-027's archetype storage and the experiment's measurements forced
attention to the per-call map-lookup tax. The decisions below codify
the API surface this session walked.

**Decision**:

1. **`entity.ID` shape**: `uint64` packed 32 bits index | 32 bits
   generation. Bevy/EnTT precedent. Initial generation is `1`;
   index 0 is reserved (never allocated). Zero-value `entity.ID(0)`
   is intrinsically invalid (index 0, generation 0; the allocator
   never produces generation 0). Packing/unpacking helpers
   (`Index() uint32`, `Generation() uint32`) live in
   `engine/ecs/entity/` per D-023.

2. **Spawn / Despawn**:
   - `Spawn(w *World) entity.ID` — allocates an EntityID
     immediately (the allocator hands it out at the call site)
     and queues a spawn for the next ApplyDeferred. The entity
     is reserved but unplaced until apply.
   - `Despawn(w *World, eid entity.ID) error` — queues a despawn
     for the next ApplyDeferred. Generation-validated; returns
     `ErrStaleEntity` synchronously on mismatch. The apply pass
     drops any queued Attach/Detach for the entity (despawn
     supersedes pending mutations on the same entity).

3. **Allocator ownership**: the `entity.Allocator` type lives in
   `engine/ecs/entity/`. Each `*ecs.World` instance holds its own
   allocator as an unexported field; the allocator is *not* a
   package-level singleton. Multiple worlds (test isolation,
   level streaming, editor preview, prediction/rollback) are
   structurally supported. There is **no public constructor** for
   `entity.ID(index, gen)` — the only legitimate way to obtain a
   valid `entity.ID` is through `Allocator.Allocate()` (or, in
   future, a deserialization helper that itself goes through the
   allocator). The type's invariant — "all live IDs trace back to
   an allocator that knows their generation" — is structurally
   enforced.

4. **Attach / Detach**:
   - `Attach[T any](w *World, eid entity.ID, value T) error` —
     queues an attach; **upsert** semantics (overwrites if T is
     already present); generic over T (CID derived via
     `component.IDFor[T]()`); takes T by value.
   - `Detach[T any](w *World, eid entity.ID) error` — queues a
     detach; **no-op if T is absent**; returns `ErrStaleEntity`
     synchronously on stale handle.

5. **Read / Write**:
   - `Get[T any](w *World, eid entity.ID) (*T, error)` — returns
     a pointer into archetype column data; no copy. `ErrStaleEntity`
     on generation mismatch; `ErrNoComponent` if the entity does
     not have T.
   - `Set[T any](w *World, eid entity.ID, value T) error` —
     copies value into the entity's component data. Convenience
     form of `*Get(...) = value`.
   - `Has[T any](w *World, eid entity.ID) bool` — true iff entity
     is live (generation valid) and has component T.

6. **Map-lookup tax fix at the storage layer.** Archetype's
   per-CID column lookup uses a flat array
   `columnFor [MaxCID]int16` (column index per CID, or `-1` if
   absent), *not* a `map[component.ID]Column`. One array index,
   no hash. Per-archetype memory cost: ~4 KB at MaxCID=2048;
   total cost sub-MB for hundreds of archetypes. Estimated
   `Get` per-call cost: ~5–8 ns (down from the experiment's ~13
   ns map-lookup tax). At 200k ops/frame this is ~1.0–1.6 ms
   instead of ~2.6 ms.

7. **Pointer lifetime**: pointers returned by `Get` (and by
   `Query` iteration callbacks) are valid **until the next
   `ApplyDeferred`**. ApplyDeferred may move entities to
   different archetypes, invalidating the pointers. Single-threaded
   per D-024; ApplyDeferred runs at known scheduler sync points
   (the exact placement is concept-tier in
   `concepts/engine/scheduler.md`). User discipline rule, not a
   runtime check — Go has no lifetime system to enforce it.

8. **Queued-despawn semantics**: a despawn that has been queued
   but not yet applied does **not** invalidate reads, queries, or
   subsequent Attach/Detach against the entity. Between
   `Despawn(eid)` and the next `ApplyDeferred`, the entity remains
   readable and queryable. Stale-handle errors fire only after
   recycle, which only happens at apply.

9. **Error model**: sentinel errors per conventions §8 (no
   allocation per call):
   - `ErrStaleEntity` — generation mismatch.
   - `ErrNoComponent` — entity lacks the requested component.

10. **Threading posture**: single-threaded today per D-024.
    Concurrent patterns remain forward-looking in
    `concepts/engine/scheduler.md` (signature-aware parallel ECS
    systems via D-016's read/write declaration surface) and
    `concepts/engine/render-thread.md` (render-thread separation,
    pipelining across frames). Subsystem-per-goroutine actor
    model was considered and rejected for inner-tier per D-028's
    Alternatives section (per-frame channel arithmetic collapses
    at scale); appropriate for outer-tier per conventions §10.

11. **Query**: **struct-based view**. The user defines a view
    struct whose pointer fields name the components the query
    iterates; a single generic `Query` is parameterized by the
    view type:

    ```go
    type Movement struct {
        Pos *Position
        Vel *Velocity
    }

    func NewQuery[V any](w *World) *Query[V]

    for eid, m := range ecs.NewQuery[Movement](w).All {
        m.Pos.X += m.Vel.X
    }
    ```

    Internally: `NewQuery[V]` reflects on V's fields once at
    construction (cached per V type); the signature (set of
    required CIDs) and field-to-column mappings are computed
    once. `All` is a range-over-func iterator (Go 1.23+) that
    declares one `var view V` on its iterator stack frame and
    reuses it across iterations — **no per-entity allocation**.
    Per-entity work matches the per-count-generic alternative:
    N pointer assignments to the view's fields. Iteration order
    is **deterministic**: archetype-table-insertion order, then
    row-insertion order within each archetype. Cached archetype
    list invalidates lazily on archetype creation (version-stamp
    check at iteration start; recompute only when stale).

**Reasoning**:

*EntityID shape* — uint64 packed 32+32 has Bevy and EnTT precedent
already cited in the entity-allocator concept; the voxel-game scale
envelope easily fits 4 billion concurrent entities with 4 billion
recycle generations per index. Single-integer ergonomics (map key,
register-fits, sortable, cheap to compare) outweigh the 4-byte
overhead vs. uint32 packing. The tighter uint32 ceilings would
demand defensive logic against generation overflow; uint64 makes
that effectively non-existent. Initial generation = 1 with index 0
reserved makes zero-value handles trivially detectable as invalid,
mirroring the `component.InvalidID = 0` convention from the
experiment.

*Empty Spawn + composition via Attach* — Go's lack of variadic
generics rules out a one-call typed `Spawn[T1, T2, ...]`. Per-count
generics (`Spawn1`, `Spawn2`, ...) were rejected (suffix
proliferation). Variadic `component.Value` was rejected (`unsafe.Pointer`
escapes through the user-facing API). Bundle-struct + reflection
was a viable but heavier option. Empty Spawn + Attach composition
keeps each call typed and minimal; the deferred-queue's apply
logic collapses spawn-then-N-attaches into a single archetype
placement, so the "N calls" cost is notional. The composition
shape is simpler at the call site than any of the alternatives.

*Allocator owned by World* — package-level singleton would be
ergonomically cleaner at the call site (`entity.New()` vs.
`ecs.Spawn(w)`) but breaks multi-world support (test isolation,
level streaming, prediction/rollback) and obscures lifecycle.
World ownership achieves the same call-site ergonomics
(`ecs.Spawn(w)` is the user-facing API regardless) without the
costs. Conventions §10's pushback on package-level globals
reinforces this. The "no public constructor for `entity.ID(index,
gen)`" rule makes the invariant structurally enforced.

*Attach/Detach as upsert/no-op* — the alternative ("error if
already present" / "error if absent") forces verbose
`if !Has[T] { Attach[T] }` patterns at every call site without
catching real bugs (the semantically interesting failure — stale
entity — still surfaces as an error). The experiment's archetype
implementation already uses upsert (R-013).

*Map-lookup tax at storage layer, not API layer* — D-027 named
this as "the engine's typed call-site surface design will want
this finding in scope." The cleanest fix is at the storage layer
(flat-array column index), keeping the API surface clean and
addressing the tax at its source. Estimated per-call cost
falls from ~13 ns to ~5-8 ns, recovering most of the gap.

*Pointer lifetime as a documentation rule* — Go has no lifetime
system; runtime checks would require allocator-validated
"smart pointers" that defeat the purpose of returning a raw `*T`.
Single-threaded execution per D-024 makes ApplyDeferred sync
points well-defined, so the rule is mechanical to follow at
the human level.

*Queued-despawn doesn't invalidate reads* — consistent with
D-024's deferred-mutation discipline (the world doesn't change
mid-frame); avoids "the entity disappeared between line 5 and
line 8 of my system" surprises; matches the experiment's
deferred-queue semantics.

*Struct-based view Query (not per-count generics)* — Go 1.25 had
no language changes; Go 1.26 added expression-arg `new` and
self-referential generic constraints, neither of which helps.
Variadic type parameters remain unimplemented in the language.
Per-count generics (`Query1`, `Query2`, ...) hit the suffix
proliferation problem. Struct-based view sidesteps the variadic
constraint entirely: the view struct's field set is the
component set; reflection extracts the signature once per view
type; range-over-func iteration with a reused view value avoids
per-entity allocation. Per-entity cost is comparable to per-count
generics (N pointer assignments either way). The view struct also
serves as a semantic unit (`Movement`, `Renderable`, etc.) that
improves codebase vocabulary.

**Alternatives** (consolidated; per-dimension rejections detailed
in Reasoning):

- *Per-count generic Query (`Query1`, `Query2`, ..., `QueryN`).*
  Type-safe at the call site, no reflection. Rejected for suffix
  proliferation; Go 1.26 confirmed no variadic-generics relief
  in sight.
- *Variadic `component.Value` Spawn / Attach.* `unsafe.Pointer`
  escapes through user-facing API. Rejected.
- *Bundle-struct Spawn (Bevy-style derive-Bundle).* Workable but
  heavier than empty-Spawn + composition; can be added later as
  an additive feature without invalidating the basic API.
- *uint32-packed EntityID (e.g., 22+10).* Tighter ceilings; demands
  generation-overflow handling. Rejected for marginal memory
  savings against the cost of defensive logic.
- *Package-level singleton allocator.* Breaks multi-world,
  test isolation, lifecycle clarity. Rejected per conventions §10
  posture and the World-as-owner principle.
- *Get/Set/Has return panics for missing components (`MustGet`).*
  Live option, but conventions §8 pushes against panics in
  production code. Deferred — can be added later as `MustGet[T]`
  if measurement or ergonomic friction justifies it; not a
  foundational requirement.

**Open questions left forward-looking**:

- *Read/write encoding in Query views (D-016 gap)*. The
  struct-based view as designed encodes the *signature* (which
  components the query touches) but not the *direction* (which
  it reads vs. writes). D-016 requires read/write at the API
  for parallel-scheduling proof. Approaches under consideration
  for the next session: marker types (`ReadOnly[Position]`),
  struct tags (`\`ecs:"read"\``), naming convention. The
  development session that lands Query design will need to
  resolve this.

- *`MustGet[T]` panicking variant.* Deferred; add if friction
  surfaces.

- *Bundle-struct Spawn convenience layer.* Deferred; add as
  additive feature if a measured ergonomic need surfaces.

- *Deserialization path for `entity.ID`.* Currently the only
  legitimate `entity.ID` source is `Allocator.Allocate()`.
  Save/load support will need a path that hands a deserialized
  ID back through the allocator — concept-tier work for when
  persistence (`engine/storage/`) is built.

- *Archetype-table memory growth past 100k entities.* Carried
  forward from D-027.

- *Threading evolution.* Carried forward from D-027 and D-028.
