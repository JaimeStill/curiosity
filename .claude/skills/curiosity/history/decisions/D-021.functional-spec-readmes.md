# D-021 — Functional-spec READMEs for infrastructure with illustrative content — 2026-05-06

**Scope**: project

**Context**: The ecs-storage-comparison experiment compares four storage-backend implementations. Comparing them — the experiment's whole purpose — requires understanding how each one shapes data and how its operations transform that shape. Source code expresses *what each operation does*; the *how the data is laid out*, *why the layout enables this algorithm*, and *what each operation transforms* dimension is illustrative content that source code expresses poorly even when well-named. After the sparsesetgroup implementation landed this session, the user requested per-backend READMEs that "describe how the implementation functions in the same way you have been illustrating how they function when I ask you as we work through the checkpoints" — pointing at the data-layout diagrams and walked-through scenarios I'd been producing during checkpoint preludes. The discipline was generalized at session closeout: this isn't ECS-specific, it applies to any infrastructure where comparing implementations or understanding data layouts is load-bearing for the work the package does.

**Decision**: For packages whose value is the way they shape data — multiple implementations being compared, or a single implementation whose data layout is non-obvious from the code alone — author an adjacent README that walks through the data model and operations using concrete examples. The README lives in the package or implementation directory it documents and carries:

- Data-model invariants — what shape is maintained, what columns/structs/maps actually contain.
- Algorithm shape under representative scenarios — concrete entity counts, ASCII layouts of buffers, walked-through state changes.
- Structural behaviors that follow from the layout — what is cheap, what is expensive, what cross-cuts.

It does *not* carry: line-by-line code paraphrase, parameter contracts, exhaustive method lists, status tables — the things godoc would carry where godoc applies. Documentation-decay still applies; sections become removable when code can fully express them.

**Reasoning**: The applicability gate — "does this documentation add signal beyond well-named code?" — is what keeps the discipline from sprawling into routine packages. A simple package whose source reads cleanly doesn't need a README; a package whose value is *how it shapes data differently from its siblings*, or whose data layout enables non-obvious algorithms, does. Experiments are the obvious case (and were the surfacing case), but the discipline is general — any infrastructure where the comparative or layout-driven dimension is load-bearing for understanding the work product benefits from this format.

The illustrative format — diagrams of what's actually in the columns, walked-through scenarios of operations transforming state — is what survives the code evolving. Field-by-field bullet lists describing structs go stale the moment a field is added; a diagram showing what a column with three entities looks like remains accurate as long as the data-model shape holds, which is the slowest-changing layer of the implementation.

Adjacency (README in the package directory, not in a centralized docs tree) ensures the documentation travels with the code through refactors and renames, and keeps update locality tight — the README is updated alongside the code it documents, in the same change.

**Alternatives**: (1) Rely on code only — rejected when the comparative or illustrative dimension is load-bearing for the work product (as it is for the storage comparison and any future comparison-style experiment). (2) Centralize all such documentation in a top-level doc tree — rejected because adjacency to the code being documented improves discoverability and ensures the doc travels with the code. (3) Treat per-package documentation as conventions-grade rather than decision-grade — rejected because the format choice (illustrative scenarios over field-bullet specifications) and the maintenance commitment are decisions worth recording, not stylistic preferences.
