# D-029 — ECS storage facade: typed direct to archetype — 2026-05-12

**Scope**: engine

**Context**: D-027 settled archetype as the engine's default ECS
storage strategy; sparsesetgroup was preserved as a per-system option
"contingent on future evidence." The experiment's `Storage` interface
(`Read(eid, cid, dst)` / `Write(eid, cid, src)` over byte slices) was
built specifically to measure multiple backends on equal terms; that
abstraction's purpose was the comparison, not engine ergonomics. With
the comparison settled, Phase B of this session opened the layering
question: does the engine retain a type-erased `Storage` interface
under a typed call-site facade (preserving D-027's swap door
explicitly), does the typed surface go directly to archetype with
no internal facade, or some hybrid?

**Decision**: The engine's typed call-site surface goes **directly
to archetype** with no internal type-erased layer. Archetype is the
engine's sole ECS world backend. The engine's `World` is concretely
backed by archetype data structures; there is no abstract `Storage`
interface in the engine's hot path.

D-027's preserved-option door for sparsesetgroup is honored
**outside the world API**, not by retrofitting a backend abstraction
across the world's hot path. If a future system surfaces a measured
need for sparsesetgroup-style storage, it lives as a *separate*
specialized storage outside the main world (the Bevy-resources
pattern: main archetype world + per-system specialized storage as
distinct APIs), not as a swapped-in backend behind the world's
typed surface.

**Reasoning**:

The experiment's `Storage` interface existed to hold the measurement
comparison together; that purpose is discharged by D-027. What
remained as an argument for retaining the abstraction in the engine
was speculative — "we might want to swap backends someday." The
project's "no premature abstractions" posture pushes back against
exactly that kind of speculative layer. D-027's exact phrasing on
sparsesetgroup is "*if such a case surfaces with measurement
justification*"; that's a future-driven option, not a present one.

The typed-vs-erased boundary is precisely where D-027's open
question on the map-lookup tax fires (the experiment's
`s.columns[cid]` map lookup surfaced as a ~13 ns/op tax in
attach/detach). A typed facade over a type-erased core has to
actively work to *avoid* re-introducing that tax at the layer
boundary. Going typed-direct sidesteps the boundary entirely;
the storage-layer fix (flat-array column index per D-030) addresses
the tax at its source.

The engine runtime touches ECS heavily (most inner-tier work runs
against ECS state); keeping the call-site surface concrete and
typed avoids paying interface-dispatch and type-erasure costs across
the broadest hot path in the engine. Concrete types and value
receivers are exactly the inner-tier divergence conventions §2
permits and encourages.

The Bevy-resources pattern (separate APIs for specialized storage
outside the main world) is a well-established escape hatch and
preserves D-027's preserved-option door without touching the
world's hot path. When and if sparsesetgroup earns its place, it
lands as a peer storage with its own typed surface, not as a
swap-in backend behind the existing surface.

**Alternatives**:

- *Type-erased core + typed facade.* Honors D-027's swap door
  directly; the typed call-site surface adapts over a clean
  abstract `Storage` interface. Rejected: more layers; the
  typed-vs-erased boundary is the home of the map-lookup tax;
  the abstraction's primary value (multiple backends) is
  speculative until a real driver surfaces. The "we might want
  to swap" framing is exactly the speculative-future posture
  the project's principles push against.

- *Typed-direct with no escape hatch.* Foreclose D-027's
  preserved-option door entirely; the engine commits to
  archetype-only forever. Rejected: throws away an option D-027
  deliberately preserved without engineering benefit beyond
  marginally simpler framing. The chosen path keeps the door
  open via per-system custom storage outside the world API.

- *Hybrid where some operations are typed-direct and others go
  through an erased interface.* Considered briefly but rejected
  as the worst of both — the layering boundary's complexity
  without its substitution flexibility.
