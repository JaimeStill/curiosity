# D-022 — Architectural alternatives weighed across performance, complexity, and ergonomics jointly — 2026-05-07

**Scope**: project

**Context**: The multi-component query measurement (R-009) showed sparsesetgroup's leadership flips between scenarios — wins ~7% over archetype on queries that match the declared owning group, trails ~17% on queries that don't. The data raised an immediate tactical question (whether to invest further effort in optimizing sparsesetgroup, e.g. splitting the unified iterator to close the fallback-path tax) and a broader strategic question (how should architectural alternatives be evaluated across the project at all). The user articulated their evaluation framework, which we both aligned on. The framing applies to engine architecture choices generally, not just the storage decision the experiment is feeding into; recording it as a project-scope decision so the criterion is durable across future architectural choices.

**Decision**: Architectural alternatives are evaluated by the joint balance of three axes:

1. **Performance** — measured throughput, latency, allocation profile, and resource use under representative workloads.
2. **Implementation complexity** — total code volume, abstraction layers, distributed invariants, surface area for subtle bugs, maintenance burden across the engine's lifetime.
3. **Development ergonomics** — how easily the implementation can be held in mind, reasoned about, extended, and debugged. How directly its abstractions map to the problem.

The "winner" is the alternative with the highest joint score across these axes, not the highest performance score in isolation. An implementation that is modestly less performant but significantly simpler and more ergonomic is preferred over an implementation that is modestly faster but significantly more complex and unintuitive. Performance gaps must be substantial and load-bearing — affecting frame fidelity at engine-relevant scales — to justify substantial complexity or ergonomic costs. The framework does not specify formal weights or scoring rubrics; the judgment is qualitative and made in context.

**Reasoning**: Complexity has carrying cost throughout the engine's lifetime. Every new abstraction layer must be maintained, debugged, and reasoned about across every future session that touches it. Performance metrics are easy to measure in a single benchmark and surface immediately; complexity costs are invisible to those benchmarks and accumulate quietly as long-term tax on every adjacent change. Without an explicit framework that names complexity as a first-class evaluation axis, the default selection criterion drifts toward "fastest wins," which over-weights micro-optimizations paying small performance dividends at large complexity cost.

Ergonomics affects the partnership-of-development directly. Code the user can hold in mind comfortably is code we can collaborate on confidently — the source-code-authorship boundary (CLAUDE.md) is sustainable only when the user can reason about what they're typing. Code that requires deep paging-in to modify creates friction across every subsequent session that touches it; ergonomics costs compound the same way complexity costs do.

The framework operationalizes the project's core principles already stated in SKILL.md — *learning over expedience*, *correctness and clarity first*, *no premature optimization*. Those principles give general direction; this decision gives the specific shape of the judgment when comparing concrete alternatives where each axis trades against the others.

**Alternatives**:

*"Fastest wins"* — rejected. Complexity and ergonomics are invisible to a pure performance metric and accumulate as long-term cost the benchmark cannot reveal. Optimizing for the easiest-to-measure axis ignores the axes that matter most for collaboration and long-term project health.

*"Simplest wins"* — rejected. Correctness-and-clarity-first does not mean abandoning performance entirely. The engine's frame budget at scale is real; performance gaps that materially affect frame fidelity must influence the decision. The framework treats performance as a peer of the other axes, not as an afterthought to simplicity.

*"Performance with complexity tiebreaker"* — rejected. Complexity and ergonomics are not tiebreakers applied only when performance is roughly equal; they are first-class axes evaluated alongside performance from the start. A "tiebreaker" framing would weight performance higher than the other axes by default, reproducing the bias this decision exists to correct.
