# D-025 — Sparse-set Despawn membership lookup: scan all columns — 2026-05-09

**Scope**: engine

**Context**: The session implementing `Despawn` + `ApplyDeferred`
across the three active backends had to settle one of R-010's open
questions for the sparse-set variants: when an entity is despawned,
how does the storage know which columns to remove it from?

`sparsesetslice` and `sparsesetgroup` keep each component column
self-sufficient — `sparse []int32` indexed by `entity.ID`, `dense
[]byte` packed values, `entities []entity.ID` parallel to dense.
There is no top-level entity → component-set map. Two options were
tractable:

- *Option A* — scan all columns at Despawn time, probing each with
  `int(id) < len(c.sparse) && c.sparse[id] != -1` to gate per-column
  swap-remove. Cost scales with `len(s.columns)`; per-probe cost is a
  bounds check plus one int32 load.
- *Option B* — maintain a top-level `signatures map[entity.ID]
  component.Signature` (or per-entity `[]component.ID`) populated by
  `Spawn` and updated by `Attach` / `Detach`. Despawn visits only the
  columns the entity lives in. Cost scales with components-per-entity;
  reintroduces centralized per-entity state.

The choice is load-bearing for the structural-churn workload's
measurements — Option B's tracking overhead vs. Option A's scan cost
will surface there for the first time.

**Decision**: Option A — scan all columns. Both `sparsesetslice.
applyDespawn` and `sparsesetgroup.applyDespawn`'s Phase B iterate
`s.columns` and probe `int(id) < len(c.sparse) && c.sparse[id] != -1`
before calling `c.swapRemove(id)`. No per-entity membership tracking
is added.

**Reasoning**: The defining design pitch of the sparse-set strategy
is that columns are independent. Option B reintroduces a centralized
per-entity index — exactly the structure archetype keeps in
`s.locations` — and softens the comparison the experiment was built
to make. Option A preserves the strategy's structural identity, and
the structural-churn workload (next session) is the measurement
designed to expose whether per-Despawn scan cost is the wrong call.

If measurements show Option A's scan cost dominates, the switch to
Option B is small and well-scoped: add the membership map, populate
it in `Spawn` / `Attach` / `Detach`, and replace the `for c := range
s.columns` loop in `applyDespawn` with a lookup. Doing that move now,
without measurement support, would be premature optimization at the
exact axis the experiment exists to settle.

**Alternatives**:

- *Option B (top-level signatures map).* Rejected as premature without
  measurement support. The next session's structural-churn workload
  will surface whether scan cost matters; if it does, Option B becomes
  the next decision with measurement justification rather than
  prediction.

- *Per-column-typed tracking (e.g., a per-column bloom filter or an
  alternative absence sentinel).* Considered briefly; rejected because
  the existing `sparse[id] == -1` sentinel already provides a
  constant-time presence probe. Layering further structure adds cost
  without addressing the same-axis question.
