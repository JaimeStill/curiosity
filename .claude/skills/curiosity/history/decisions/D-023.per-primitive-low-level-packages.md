# D-023 — Per-primitive low-level packages — 2026-05-08

**Scope**: project

**Context**: While restructuring the ECS storage comparison experiment to
introduce an entity allocator (driven by D-019's measurement-validity
discipline — without recycling, structural-churn measurements would
not reflect how storage implementations behave in practice), the
natural placement question surfaced. Initial sketch: `entity.Allocator`
returns `storage.EntityID`, which would have meant `entity` imports
`storage` for the type. The user redirected: shift the entity-related
primitives down into `entity/` so `storage/` depends on `entity/`, not
the other way around. The framing extended beyond the immediate
allocator question — each core ECS primitive should own its own
low-level package, establishing the habit of clean per-primitive
ownership before more code accretes.

**Decision**: Each core ECS primitive lives in its own low-level
package that depends on nothing experiment- or engine-internal:

- `entity/` owns the entity-identity primitive (`entity.ID`) and the
  entity allocator (`entity.Allocator`).
- `component/` owns the component-identity primitive (`component.ID`),
  the component-value carrier (`component.Value`), the type→ID
  registry, and the component-set primitive (`component.Signature`).
- Higher-level packages (`storage/`, the backends, `workload/`,
  `main.go`) depend on `entity/` and `component/` as needed. The
  dependency graph is strictly one-way; no cycles. `entity/` and
  `component/` do not depend on each other.

The decision applies first to the experiment but the pattern is
intended to carry into engine architecture: each ECS primitive in
its own package, dependents above, no primitive importing another
primitive.

**Reasoning**: Establishing the boundary now, when the experiment is
small and easy to restructure, avoids discovering the layering
mistake under load. The "package per primitive" pattern keeps each
primitive's design self-contained — the entity allocator's design
lives entirely within `entity/` and is testable, swappable, and
replaceable without touching storage. It also makes dependency
direction visible at a glance: a package that imports `entity/`
consumes the entity-identity surface and nothing more.

The placement also satisfies the existing concept-tier guidance in
`concepts/engine/entity-allocator.md`: "Allocator above storage layer
... iteration code in storage backends should not be entangled with
free-list logic ... keeps storage backends comparable on equal terms
(each measured against the same allocator behavior rather than its
own private ID-allocation shortcut)." A sibling package realizes that
intent more cleanly than embedding in `storage/` would have.

**Alternatives**:

- *Embed allocator inside `storage/`.* Smaller diff, closer to status
  quo. Rejected: muddles `storage/`'s purpose (component storage and
  structural-mutation contract) by adding entity-allocation
  responsibilities, and forces backends to interpret a hypothetical
  `storage.Allocator` as both a storage-layer and an entity-identity
  concern. The concept document already named the layering it wanted;
  the embed would have been weaker on its own terms.

- *Allocator as separate package but `EntityID` stays in `storage/`.*
  Allocator imports storage for the type. Avoids moving the EntityID
  type. Rejected: this puts entity semantically below allocator
  (which depends on storage for the ID type), inverting the layering
  the concept wanted. Also leaves `storage/` carrying an
  entity-identity responsibility it shouldn't own.
