# D-031 — Engine in-tree during prototype phase — 2026-05-13

**Scope**: project

**Context**: D-028 framed the engine as a sibling Go module under
`~/code/curiosity/`, with its own git history, gitignored from the
planning superproject. The scaffolding session that produced the
first engine source code surfaced that this framing is premature for
the prototype phase: the engine has no stable surface yet, infrastructure
benefits from co-location with the planning workspace, and the
overhead of cross-repo coordination outpaces the value of separation
while the engine churns. The user clarified mid-session that the engine
stays in-tree as a self-contained sub-module within the curiosity
repository until it reaches a release phase, at which point the lift
to its own repository per D-028's original framing is the path forward.

**Decision**: During the prototype phase, the engine lives in-tree
under `~/code/curiosity/engine/` as a self-contained Go sub-module
of the curiosity repository. Concretely:

1. **Module path**: `github.com/JaimeStill/curiosity/engine`. The
   engine module sits at the `engine/` sub-path of the curiosity
   GitHub repository, not as a separate `curiosity-engine` repo.

2. **`go.work` location**: `~/code/curiosity/go.work` (curiosity
   repository root). Standard Go-workspace topology with the
   workspace root above the modules it enrolls. Ready to enroll
   outer-tier and game modules as sibling directories under
   `curiosity/` when they land.

3. **`.gitignore`**: `/engine/` is **not** gitignored. The engine
   source is tracked in the curiosity repository's git history during
   the prototype phase. (Game and future outer-tier-implementation
   trees remain gitignored until each independently graduates to
   in-tree tracking.)

4. **D-028's sibling-repo framing applies at release.** When the
   engine reaches release stability, the lift to its own repository
   per D-028's original framing — separate `git init`, separate
   history, gitignored from curiosity — becomes the action. The
   release transition is itself a future bookkeeping event recorded
   when it happens; the destination is named here so the path is not
   re-decided then.

This decision **supersedes** D-028 on the module-boundary question
specifically (points 1, 2 of D-028's Decision section). The remainder
of D-028 — per-concern outer-tier contract packages, herald-lifted
lifecycle pattern, ECS sub-package decomposition, iterative-depth
applied to other inner-tier subsystems, naming reconciliation,
runtime at engine root — stands unchanged.

**Reasoning**: The prototype phase prizes single-repo affordances —
one commit graph, one set of CI hooks, one PR workflow, one branch
to coordinate planning-surface and source-code changes. D-028's
sibling-repo framing was correct as an end state but optimistic
about the velocity at which the engine would stabilize; for an
engine surface that is still surfacing its first source file, the
overhead of cross-repo coordination is pure cost.

The phase boundary itself (prototype vs. release) is the natural
discriminator: the engine warrants its own repo when its surface is
stable enough that breaking changes are rare and external consumers
benefit from versioned releases. Neither condition holds while the
engine is being scaffolded; both will hold by release.

**Alternatives**:

- *Apply D-028 as written from day one.* Rejected for the cost-
  benefit reasoning above. Premature isolation of an unstable surface.

- *Module path `github.com/JaimeStill/curiosity-engine`.* Considered
  during the session as the literal mirror of herald's
  `github.com/JaimeStill/herald` pattern. Rejected because the path
  implies a separate `curiosity-engine` repository on GitHub, which
  is not the prototype-phase reality. The chosen path
  (`github.com/JaimeStill/curiosity/engine`) is honest about the
  in-tree layout.

- *Leave `go.work` at `~/code/curiosity/engine/` per R-014's literal
  "engine root" phrasing.* Rejected because once outer-tier and game
  modules land as siblings under `curiosity/`, the workspace root
  needs to span all of them. Placing `go.work` at the curiosity
  repository root from the start avoids a later relocation. R-014's
  "engine root" phrasing is reinterpreted as "the root that owns
  the engine," which under this decision is the curiosity repository
  root.
