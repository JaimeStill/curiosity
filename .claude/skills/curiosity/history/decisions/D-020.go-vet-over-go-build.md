# D-020 — go vet preferred over go build for source verification — 2026-05-06

**Scope**: project

**Context**: During the sparsesetgroup iteration-baseline session, I proposed `go build ./...` as the verification step before running the experiment harness. The user redirected to `go vet ./...`. `go vet` runs the typecheck pass that `go build` runs, plus a suite of static-analysis passes (printf format mismatches, accidentally-copied locks, malformed struct tags, unreachable code, and others). For the verification use case — confirming source is well-formed before running it or committing it — vet is strictly stronger than build. The previous behavior-layer guidance in `verification.md` listed both commands as available toolchain checks without ranking them; the redirect surfaced the gap.

**Decision**: Use `go vet ./...` rather than `go build ./...` for any source-verification step. `go build ./...` is invoked only when the intent is producing a binary artifact (release builds, distribution, packaging). `go test ./...` remains the right choice for running unit tests. The operational rule lands in `.claude/behavior/verification.md`; this entry captures the reasoning.

**Reasoning**: Source verification's purpose is catching errors before they reach runtime or commit. `go vet` catches everything `go build` catches (since vet runs the build's typecheck pass internally) plus a class of bugs `go build` ignores. The additional passes flag exactly the kind of bug that compiles cleanly but produces wrong behavior at runtime: `Printf("%s", n)` formatting a number as a string, a `sync.Mutex` accidentally copied into a struct field, a struct tag with malformed JSON syntax. Catching them at the verification step keeps them out of test runs and out of source-of-truth commits. The cost is no greater than `go build` (vet runs the build pass internally), so there's no reason to prefer the weaker check.

`go build` remains load-bearing when the intent is producing a binary. CLI tools, release pipelines, and distribution workflows still call `go build`; verification, where any binary produced is throwaway, should not.

**Alternatives**: (1) Continue using `go build` for verification — rejected because it's strictly weaker than vet for this purpose and offers no compensating benefit. (2) Run both `go build` and `go vet` — rejected because the build pass runs inside vet anyway; explicit double-running is redundant. (3) Treat the choice as situational — rejected because the verification step is invoked frequently across sessions, and an unranked "use whichever" leaves the slot for the weaker check to creep back in.
