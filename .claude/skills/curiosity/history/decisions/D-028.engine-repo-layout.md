# D-028 — Engine repo layout — 2026-05-12

**Scope**: project

**Context**: R-013's "Next session focus" named scoping the engine
source repo and the first piece of engine code as the work for the
context session that produced this decision. Phase A of that session
walked the layout dimensions left open by prior decisions: module
count and boundary axis, inner-tier package decomposition (where
archetype storage lives per D-027, where the entity allocator lives,
where the runtime lives), outer-tier posture, and the plug-in
mechanism between the runtime and outer-tier modules. The conventions
doc had settled the workspace topology at the meta level
(multi-module workspace; `go.work` at engine root; engine modules
under `~/code/curiosity/engine/`; conventions §3) but module
decomposition and package layout were not committed. D-023 had
established per-primitive packages for `entity/` and `component/`
"intended to carry into engine architecture"; the engine layer needed
to confirm that pattern alongside the rest of the structural
commitments.

**Decision**:

1. **Engine is a single Go module** under `~/code/curiosity/engine/`.
   No sub-modules inside the engine repo at this stage; if a future
   subsystem accretes a heavy third-party dependency (the conventions
   §3 sub-module rationale), the split lands then with measurable
   justification.

2. **Outer-tier implementations are sibling modules** under
   `~/code/curiosity/`, each with its own git history. The game is
   also a sibling module under `~/code/curiosity/`. The curiosity
   workspace repo is the planning superproject; the engine,
   outer-tier-implementation, and game repos are gitignored from it.

3. **Per-concern outer-tier contract packages live inside the engine
   module**: `engine/audio/`, `engine/network/`, `engine/storage/`,
   `engine/content/`. Each package defines the contract (interface
   plus bridge code) for its concern; outer-tier implementations
   (separate sibling modules) satisfy the contract. The interface
   lives with its consumer (the engine), not its producer
   (conventions §5).

4. **Lifecycle plug-in mechanism is the herald-lifted Coordinator
   pattern.** An `engine/lifecycle/` package holds the Coordinator
   (lifted from `~/code/herald/pkg/lifecycle/lifecycle.go` with
   adaptations for the engine's frame-loop context). Outer-tier
   contracts include a `Start(*lifecycle.Coordinator) error` method;
   the binding is uniform across concerns. Two-phase initialization:
   `engine.New(cfg)` is cold start (dependency tree assembly,
   wireability validation — the analog of herald's `/healthz`);
   `engine.Start()` (or `Run()`, name deferred to development
   session) is hot start (subsystems online; frame loop ready — the
   analog of `/readyz`).

5. **ECS sub-packages landing now**: `engine/ecs/entity/`,
   `engine/ecs/component/`, `engine/ecs/archetype/`. This carries
   forward D-023's per-primitive pattern into the engine layer.
   `engine/ecs/system/` is *not* introduced now — its primitives are
   concept-tier in `concepts/engine/scheduler.md` and the package
   awaits the session that designs them.

6. **Iterative depth applied to remaining inner-tier subsystems.**
   `engine/voxel/`, `engine/physics/`, `engine/render/`, and
   `engine/ecs/system/` are known-to-exist eventually but are *not*
   scaffolded until the work that populates them begins. Premature
   sub-packages would carry empty or thin contents that violate the
   design/concepts discipline.

7. **Naming reconciliation across "Task" and "System".** D-017's
   reservation of `System` for non-ECS use was too restrictive;
   Go's package namespacing handles the ambiguity. The reconciled
   convention:
   - `Task` (D-016 / D-017) is the engine-level scheduled-work
     primitive — *not* an ECS-specific concept. It lives wherever
     the scheduler concept eventually settles.
   - `System` is valid in two namespaced contexts: `engine/ecs/system/`
     (the S of ECS, per standard ECS triad) when that package
     lands, *and* engine-subsystem interface naming. The choice
     between package-as-name (`audio.Audio`, `storage.Storage`) and
     generic `System` interface naming (`audio.System`,
     `storage.System`) is open and deferred to the sessions that
     design those interfaces.

8. **Runtime placement: engine root package.** The `Engine` type,
   the constructor, and the frame loop live at `engine/` (the
   module's root package). No `engine/runtime/` sub-package — the
   runtime's responsibilities decompose into other sub-packages
   (lifecycle → `engine/lifecycle/`; scheduling → eventually
   `engine/scheduler/` or absorbed into ECS; resource ownership →
   `Engine` struct fields; frame loop → root package methods).
   What looks like runtime weight in `design/engine/runtime.md`'s
   Runtime roles section either decomposes into its own sub-package
   or is small enough to live at the root.

**Reasoning**:

*Single engine module* — conventions §3 cites sub-modules as a tool
for isolating heavy third-party dependencies. No inner-tier
subsystem has such a dependency, nor is one anticipated. Splitting
later when a real driver surfaces is straightforward; splitting now
buys hypothetical isolation against work that hasn't started.

*Outer-tier and game as siblings* — the tier boundary in D-002 is
where stable contracts apply. Making that boundary a *module*
boundary reinforces it structurally — an outer-tier module
implements the engine's contract via a regular Go import and cannot
reach across the module boundary into engine internals through
ordinary means. The game is structurally a consumer module (imports
engine + chosen outer-tier modules), distinct from outer-tier
modules functionally even though both live as siblings.

*Per-concern outer-tier contract packages* — interface-at-consumer
(conventions §5) places the contract in the engine, alongside the
bridge code that consumes it. A centralized `engine/outer/` or
`engine/contract/` umbrella was considered and rejected: it
fractures the natural cohesion between each contract and its
bridge code, and creates a "god package" of unrelated concerns
whose only common property is "outer tier."

*Herald lifecycle pattern* — herald's Coordinator is small (~80
LOC), proven in production, and shaped exactly to the cold/hot
split that an engine frame-loop initialization needs. Lifting it
gives a known-good starting point; the adaptations (frame-loop
drive instead of HTTP request loop) are concept-tier work in
`concepts/engine/outer-tier-contract.md`. The two-phase init
maps directly onto the engine's bring-up sequence: dependency
wiring before subsystems are online; subsystems brought online
before the frame loop ticks.

*ECS sub-packages now / others later* — D-023 already settled the
per-primitive pattern for `entity/` and `component/` with the
explicit intent to carry into engine architecture. Archetype is
the storage layer per D-027 and earns its own sub-package as a
peer to entity/component. The remaining sub-packages (system/,
voxel/, physics/, render/) are known-to-exist but the work that
populates them hasn't begun, and SKILL.md's iterative-depth
principle is explicit that depth without imminent work is the
same trap as premature implementation.

*Naming reconciliation* — D-017's reasoning ("System collides with
engine's existing system usage") underestimated Go's package
namespacing. `ecs.System` and `audio.System` are distinct types;
the collision D-017 feared doesn't materialize at the language
level. D-017's substantive content (Task is the engine's
scheduled-work primitive, with subtypes per D-016's cadence axis)
is preserved; the reservation framing is loosened. Engine-subsystem
interface naming convention is intentionally not decided here —
choosing between `audio.Audio` and `audio.System` is something
to settle when the first concrete subsystem interface is designed,
with the concrete interface shape in view.

*Runtime at engine root* — every responsibility named in
`runtime.md`'s Runtime roles either decomposes (lifecycle →
its own package; scheduling → its own concept-tier home;
resource ownership → struct fields) or is small (frame loop). A
dedicated `engine/runtime/` package would mostly hold the Engine
struct and its constructor, which read more naturally at the
module's root.

**Alternatives**:

- *Multi-module engine (split by tier or per inner-tier subsystem).*
  Maximally decoupled, expensive in version-drift management.
  Rejected on conventions §3 grounds (no driver) plus the inner
  tier being *defined* by shared memory layout (D-002) — splitting
  it across modules complicates the invariant the inner tier exists
  to maintain.

- *Centralized outer-tier contract package
  (`engine/outer/` or `engine/contract/`).* Considered and
  rejected during Phase A discussion. Fractures cohesion between
  each contract and its bridge code; creates an artificial
  umbrella whose contents have no shared concerns beyond "outer
  tier."

- *Subsystem-per-goroutine actor model for inner-tier.* The user
  asked whether each engine subsystem could run on its own
  goroutine with channel-mediated communication. Rejected for
  inner-tier per D-002's tier criterion: inner-tier subsystems
  share per-entity per-frame data; per-call channel ops at ~100 ns
  collapse the frame budget at scale (~2.4M channel ops/sec for
  physics↔ECS traffic alone at 10k entities × 60 fps × 4
  ops/entity, before any actual work). The actor model is the
  *outer-tier* pattern per conventions §10; inner-tier
  multi-threading happens through within-subsystem parallelism,
  scheduler-aware parallel ECS systems, and render-thread
  separation — all concept-tier in `concepts/engine/scheduler.md`
  and `concepts/engine/render-thread.md`.

- *Generic `System` interface as standard for engine subsystems.*
  Live option but not committed; deferred to when subsystem
  interfaces are concretely designed. Package-as-name (`audio.Audio`,
  `storage.Storage`) is the other live option.

- *`engine/runtime/` sub-package.* Considered. Rejected because
  the runtime's responsibilities decompose into other sub-packages
  or are thin enough to live at the engine root; the sub-package
  would mostly hold the Engine type and its constructor.
