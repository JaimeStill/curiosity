# D-033 — Experiments as historical artifacts — 2026-05-13

**Scope**: project

**Context**: SKILL.md's Experiments section settled experiments as
time-bounded — "An experiment exists only while its job is active"
— with explicit removal triggers on both success path (removed at
the integration session's closeout) and failure path (removed once
the finding is captured). The framing followed from the
working-tree discipline that governs `design/` and `concepts/`:
only forward-looking material in the tree, completed material in
git history. At the closeout discussion of the ecs-entity-primitives
session, evaluating `experiments/ecs-storage-comparison`'s state
surfaced the broader observation: experiments under `experiments/`
are structurally isolated from engine and game code, conventions
already exempt them (D-012), and a learning project benefits from
the investigative arc being visible as reference material rather
than archived to git history where archaeology has nontrivial cost.

**Decision**: Completed experiments are retained under
`experiments/` as historical artifacts. The README's *Finding*
section captures the experiment's durable conclusion; the directory
persists past the milestone that would have triggered removal under
the prior policy.

**Reasoning**:

*Isolation already structural.* `experiments/` is its own tree
sibling to source. Experiments do not import from engine or game
modules and are exempt from `code/conventions.md` per D-012. The
isolation that justifies exempting experiment code from project-wide
conventions also justifies keeping completed experiments visible —
they cannot drift into production code, only inform it.

*Reference value in a learning project.* The investigative arc of
an experiment — Question, Approach, Finding — is itself a record
of the project's reasoning. Removing the experiment removes the
narrative; recovering it requires git archaeology that is costlier
than the tree presence. The README format is designed to read as
a coherent artifact, not a transient scratchpad.

*Decision-log entries are not substitutes.* A decision log captures
*what* was decided and *why* in summary form. An experiment captures
*the measurement that informed the decision* in much more detail
than the decision entry carries. Both are durable; they record
different layers of the same reasoning.

*Prospective application.* The policy applies to experiments
completing under the new framing. Experiments whose READMEs
explicitly cite the prior removal-at-integration trigger (e.g.,
`ecs-storage-comparison`) keep their existing language as part of
their historical record; the new policy supersedes the trigger
without requiring retroactive README revisions.

**Alternatives**:

- *Keep the time-bounded framing.* Carries the
  working-tree-only-forward-looking-material principle through
  cleanly. Rejected: the principle was developed for `design/`
  and `concepts/`, where the cost of stale forward-looking text
  is real (a reader cannot tell what is current). Experiments do
  not have the same failure mode — their state is named by the
  README's Finding section, not by their presence in the tree.
- *Move completed experiments to an `experiments/archive/`
  subtree.* Preserves visibility while explicitly signalling
  "completed." Rejected for ceremony cost — the README's Finding
  section already signals completion, and the move-on-completion
  step is an extra ritual with no functional payoff.

**Implications**:

- SKILL.md Experiments section revised: "time-bounded" framing
  replaced with "historical artifacts"; Success and Failure paths
  no longer end in removal.
- SKILL.md Reset Protocol *Culled* bullet revised: experiments
  dropped from the enumeration of what *Culled* covers.
- SKILL.md Reset transaction summary shape revised: the *Culled*
  field description loses "(includes completed experiments)".
- The no-graduation rule (experiment code never becomes engine or
  game code) is unchanged — that rule governs implementation, not
  retention.
