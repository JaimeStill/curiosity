# D-026 — Archetype `locations`: map → ID-indexed slice — 2026-05-09

**Scope**: engine

**Context**: The structural-churn workloads measured at 1k/10k/100k
surfaced archetype as the slowest backend on every structural workload
at 100k by 40-60% over `sparsesetslice` (e.g., `structural_growth`
p50: archetype 314,887 ns vs sparsesetslice 190,004 ns). Archetype's
peak heap at 100k on `structural_growth` was 143 MB versus
sparsesetslice's 93 MB — a ~50 MB gap consistent with archetype's
`locations map[entity.ID]location` carrying ~80 bytes/entry of map
internal overhead at ~1M live entries. R-008's iteration-baseline
finding had already credited archetype's `locations` map with ~4.8 MB
at 100k; structural workloads exercise it on every spawn, surfacing
both the heap and per-spawn cost the iteration loop never paid.

`sparsesetslice` already uses ID-indexed `[]int32` for its analogous
sparse mappings without encountering this cost. The same swap is
admissible for archetype because the engine-side entity allocator
(`concepts/engine/entity-allocator.md`) commits to dense IDs that
both representations rely on.

**Decision**: Replace `archetype.Storage.locations` with `[]location`
indexed directly by `int(id)`. Sentinel `arch == nil` marks empty
slots. `slices.Grow` handles capacity expansion when a spawned
entity's ID exceeds current slice length. `applyDespawn` writes
`location{}` to clear the slot rather than calling `delete`.

**Reasoning**: Closes the measured 100k structural-mutation gap
directly. Post-pivot p50 timings (R-012 Finding) flip the 100k
structural picture: archetype now leads sparsesetslice by 18-30%
on all three structural workloads. Heap at 100k drops 24-42% across
iteration, multi-component, and structural workloads — into parity
with sparsesetslice. Iteration and multi-component p50 timings change
within noise (~0-2%), confirming the swap doesn't regress hot-path
cost.

The swap aligns archetype's representational discipline with
sparsesetslice's: both backends now commit to dense entity IDs, both
index per-entity state directly without hashing. The dense-ID
assumption — already implicit in sparsesetslice's design — is now
load-bearing for archetype too.

`slices.Grow`'s append-amortized growth shape matches the existing
growth shape of archetype's per-archetype row buffers and `entities`
slices, so the storage's growth profile is now uniform across its
internal data structures rather than mixing append-amortized
slice-grow with map's bucket-rehash.

**Alternatives**:

- *Keep the map; pre-size or tune.* Rejected. Pre-sizing addresses
  amortized cost but not per-element overhead (~80 bytes vs slice's
  16). The structural cost is in the data structure choice, not its
  initial sizing.

- *Hybrid: map for lookup + slice for iteration cache.* Rejected as
  unnecessary complexity. The slice alone supports both access
  patterns — direct index lookup and (if iteration ever needs it)
  range traversal — without retaining the map's overhead.

- *Move `archetypes map[component.Signature]*archetype` to a slice in
  the same swap.* Considered and deferred. The `archetypes` map is
  small (one entry per distinct component signature, K ≤ ~100 in
  realistic games) and is not measured as a hot-path cost. Linear-scan
  signature lookup over a slice would only marginally improve Spawn's
  signature lookup and would add a "K stays small" architectural
  commitment without measurement support. The locations swap is
  performance-driven against measured data; the archetypes swap would
  be coherence-driven without comparable evidence.
